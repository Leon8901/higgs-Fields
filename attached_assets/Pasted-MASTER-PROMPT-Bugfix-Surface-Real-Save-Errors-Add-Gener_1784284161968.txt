## 2. Add a staleness timeout to generation polling (`artifacts/api-server/src/lib/generation/statusSync.ts` / `backgroundPoller.ts`)

Currently, if an async generation's poll keeps failing (network issue, transient auth error, rate limit, unexpected response shape), `syncGenerationStatus`'s catch block logs it and leaves the generation in `pending`/`processing` forever — no timeout, no eventual failure, no refund. A user can be stuck looking at "Processing..." indefinitely with zero explanation, even if the underlying provider task actually finished or errored out on their side and your poll simply can't reach or parse it anymore.

Add a staleness check: if a generation has been in `pending`/`processing` for longer than a reasonable ceiling (e.g. 10 minutes — adjust based on your longest real video generation time, don't guess a number shorter than your slowest legitimate job), the background poller should call `finalizeGeneration` with a `failed` result and a clear message ("This generation timed out — please try again"), which already triggers the existing credit-refund path in `finalizeGeneration`. Use `generation.createdAt` (or a dedicated `startedAt` if that's more accurate) to compute elapsed time — check the `generations` schema for the right column before adding a new one.

This does not replace debugging any *specific* current stuck generation — it prevents the *next* one from being stuck forever with no resolution. For the generation already stuck right now, check the Replit server console for a `"Provider poll failed"` log line tied to its ID — that will show the real underlying error (auth, rate limit, or an unexpected response shape) and tell us whether anything beyond this timeout fix is needed.
