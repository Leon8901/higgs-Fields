## Rule zero, before anything else

You cannot tell from a screenshot whether a button, toggle, or field is real or decorative. Neither can the person reviewing your work. **Phase 1 below is mandatory and must be done, and its findings reported in writing, before you change a single line of code.** If Phase 1 finds something is already real and working, leave it alone — don't rewrite working code to match this spec's wording if the underlying behavior is already correct. If Phase 1 finds something that looks real but isn't wired up, that's exactly what Phase 3 onward exists to fix.

No hardcoded values. No placeholder UI that isn't connected to a real backend call. No claiming something works because the code "looks right" — only because you clicked it, watched the network tab, and confirmed it survives a page reload.

## PHASE 4 — Platform Providers (`/admin/providers/platform`)

Developer/owner-only: the platform's own API keys, used to fund generation for users without their own BYOK key.

**4a. Real audit before building/fixing anything here**: confirm the actual current provider rows (`SELECT slug, status FROM providers`), which have a real code adapter (check `registry.ts`'s adapters map), and real model counts (`SELECT adapter, COUNT(*), COUNT(*) FILTER (WHERE is_active) FROM models GROUP BY adapter`). Report these numbers. A provider row with no code adapter can store a platform key for later, but can't be tested or enabled for live routing — the UI must say so honestly.

**4b. Schema** — extend the existing `providers` table (don't create a parallel table):
```
platform_enabled        boolean, not null, default false
encrypted_platform_key  text, nullable   -- AES-256-GCM via lib/crypto.ts, same scheme as user_api_keys
platform_key_last_four  text, nullable
platform_key_status     text, nullable   -- "valid" | "invalid" | "unknown"
platform_key_validated_at  timestamp, nullable
base_url                text, nullable
metadata                jsonb, not null, default '{}'
sort_order               integer, not null, default 0
```
Do not touch or duplicate `status` — that field already controls BYOK availability in two existing files. `platform_enabled` is a separate, distinct concept. Backfill `wavespeed`'s row with its existing `WAVESPEED_API_KEY` env var so the one provider that currently works keeps working identically.

**4c. Honest model counts, three tiers, never fabricated**:
- Cataloged / Enabled — real `COUNT(*)` from your `models` table per adapter, always shown, including zero.
- Live Available — only for adapters with a real, confirmed discovery endpoint. OpenAI's `GET https://api.openai.com/v1/models` is real and confirmed working — implement an optional `listAvailableModels?()` adapter capability (same pattern as the existing optional `listVoices?()`) for it. For any provider without a genuine discovery endpoint, show "Live catalog not available for this provider" — never a guessed number.

**4d. Routes** (new file, owner-gated): `GET /admin/providers` (all rows + `hasAdapter` + counts + masked key info), `PATCH /admin/providers/:slug` (rejects `platformEnabled: true` for a slug with no adapter, with a clear message), `POST /admin/providers/:slug/platform-key` (validate via `adapter.validateKey` if it exists, encrypt, store), `DELETE .../platform-key`, `POST .../test-connection` (real pass/fail or an honest "not testable" reason — never a fake green check), `POST .../icon` (upload or paste-URL via the Phase 6a shared utility).

**4e. Rewire the actual generation key routing** — `getPlatformApiKey()` in `keyRouting.ts` is a hardcoded switch statement today with one working case. Replace with a real async DB lookup gated on `platformEnabled` + `encryptedPlatformKey`. Update its call site to `await` it. **This touches your only currently-working payment path (WaveSpeed)** — after this change, run one real WaveSpeed generation end-to-end and confirm it still succeeds before considering this phase done.

**4f. Frontend**: render provider cards by mapping over the real array `GET /admin/providers` returns — `providers.map(provider => <ProviderCard {...provider} />)` — do NOT write a separate hardcoded component or JSX block per known provider name (no `<OpenAICard />`, no if/else branching on `slug === "openai"`). Every field on the card (name, icon, `hasAdapter` badge, masked platform key status, Enable/Disable toggle, Test Connection, Cataloged/Enabled counts, Live Available or the honest fallback label) comes from that one row's data. This is what makes adding a new provider later a database insert, not a frontend code change.

## PHASE 6 — Object storage: diagnose for real, then fix

**A real upload failure was already observed**: `HTTP 500 — Failed to generate upload URL`. The route currently swallows every possible cause into that one generic message:

```javascript
} catch (error) {
  req.log.error({ err: error }, 'Error generating upload URL');
  res.status(500).json({ error: 'Failed to generate upload URL' });
}
```

**Before writing any fix, check the actual Replit server console log for the real underlying error** — it's already being logged via `req.log.error`, just not surfaced to the browser. Most likely cause given this project's documented history of secrets being wiped on re-import: `PRIVATE_OBJECT_DIR` isn't set in this environment. Confirm via the Replit Secrets panel and the Object Storage tool's provisioning state — don't guess, check.

**Fix the error handling regardless of the root cause**, so future failures are self-diagnosing instead of requiring a server log lookup:
```javascript
} catch (error) {
  req.log.error({ err: error }, 'Error generating upload URL');
  const message = error instanceof Error && error.message.includes('not set')
    ? 'Object Storage is not configured on this server — check PRIVATE_OBJECT_DIR.'
    : 'Failed to generate upload URL';
  res.status(500).json({ error: message });
}
```

**6a. Shared asset-import utility** (`artifacts/api-server/src/lib/media/assetImport.ts`): given a URL, fetch with a timeout, verify a real image `Content-Type`, enforce a max size (~5MB), upload to object storage, return the owned path. Used by Site Branding's Logos & Icons tab AND both Providers pages' icon fields — write once, call three times.

**6b. Real health check** — `GET /healthz` is currently a hardcoded `{ status: "ok" }` stub with no actual check behind it. Extend it (or add an owner-gated `GET /admin/settings/health`) to run a real DB query and a real check that `PUBLIC_OBJECT_SEARCH_PATHS`/`PRIVATE_OBJECT_DIR` are set, and report the real result — this is what backs the Platform Status card's Database/Object Storage rows. If either check fails, that card must visibly show "Disconnected," not stay hardcoded green.

**6c. Test both upload paths**: inside the embedded Replit preview iframe, and in a real new browser tab. Report which one actually works — if a real new tab still fails, that confirms a genuine backend problem (likely Phase 6's env var issue), not an iframe sandboxing artifact.
