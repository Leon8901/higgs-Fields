## 2. Backend routes (`artifacts/api-server/src/routes/admin-models.ts`, new file, owner-gated)

- `GET /admin/models` — returns every row from `models`, including `thumbnailUrl`, `isActive`, `adapter`, `category`, `name`, `creditCost`, and whatever other columns already exist on the table — check the actual schema, don't assume a field list.
- `PATCH /admin/models/:modelId` — updates editable fields: `thumbnailUrl`, `isActive`, `name`, `description`, `badge`, `sortOrder`, `creditCost` (or the real equivalent column names — check the schema first, don't guess). - `POST /admin/models/:modelId/thumbnail` — accepts either a file upload (presigned PUT, same flow already used for logos/favicons/provider icons) or `{ url: string }` for paste-URL. **Reuse the exact same shared `assetImport.ts` utility already built** (download-and-reupload with content-type/size validation, never store a raw external URL) — this is the third consumer of that utility (after Site Branding logos and Provider icons), not a fourth reimplementation.

Add to `openapi.yaml`, run codegen, per existing project convention.

## 3. Frontend — data-driven, not hardcoded per model

Render model cards by mapping over the real `GET /admin/models` array — `models.map(model => <ModelCard {...model} />)`, same rule as the Platform Providers/BYOK Providers pages already built. Do NOT hardcode a card per known model name.

**Page toolbar, at the top**:
- **Search bar** — client-side filter over the already-loaded models array, matching against `name` (and `description` if useful) as the admin types. No API call per keystroke — filter the data already in memory, same pattern as Site Branding's settings search.
- **Provider filter** — a dropdown or button group listing every distinct `adapter`/provider value actually present in the loaded models (derive this list from the real data — `[...new Set(models.map(m => m.adapter))]` — do NOT hardcode a fixed list of "OpenAI, Kling, Claude, Google" options, since that list must automatically include whatever providers actually have models today and never show a provider with zero models, or omit one that has models but wasn't in a hardcoded guess). Selecting a provider (e.g. "OpenAI") filters the visible cards to only models where `adapter === "openai"`. Selecting "All" (default) shows everything. Search and provider filter combine — e.g. searching "pro" while filtered to OpenAI shows only OpenAI models with "pro" in the name.
- Category filter (image/video/audio/text) as a second, independent filter alongside the provider filter — same real-data-derived approach, not hardcoded.

Each card, styled similarly to the reference screenshots (image thumbnail as the dominant visual element, category badge, name, short description) but wired to real data and real actions:
- Thumbnail preview — shows the current `thumbnailUrl` if set, or an honest empty/placeholder-icon state if not set (a generic "no thumbnail" icon state is fine and is NOT the kind of placeholder this task is trying to eliminate — that term refers to fake data pretending to be real, not an honest empty state for a field that's genuinely unset).
- **Upload new** and **Paste URL** buttons for the thumbnail, both calling the Section 2 thumbnail endpoint.
- **Editable name** — inline edit or a small edit panel, wired to the PATCH route, saved for real.
- Active/Inactive toggle (`isActive`), wired to the PATCH route.
- Other basic editable fields (description, sort order, credit cost) if they already exist as real columns — inline edit or an expand/edit panel, whichever fits the existing design pattern from the Providers pages.
